# v0.3.1 — Docs & Discovery Optimization

**Date:** 2026-04-23
**Baseline:** v0.3.0 (tag `817ded9`), 62 tests passing, `flutter analyze` clean, `flutter pub publish --dry-run` 0 warnings.
**Scope:** Documentation, metadata, and example only. NO changes under `lib/`.

## Version Strategy: Publish v0.3.0 First, Then Ship v0.3.1

**Decision:** Publish v0.3.0 to pub.dev as-is (0 warnings, tests green), then bump to v0.3.1 for this docs pass.

**Rationale:**
- v0.3.0 contains meaningful API additions (abortSnap, pinnedDivider). Delaying publish to bundle docs polish blocks users who need the feature now.
- A docs-only 0.3.1 is a clean semantic signal: "nothing changed in the code, we just improved onboarding." This is a pub.dev convention for docs releases.
- If we amend v0.3.0 instead, we either (a) force-yank and re-publish (bad practice, pub.dev discourages it) or (b) never publish the clean v0.3.0, losing the clean checkpoint.
- Sequence: `flutter pub publish` v0.3.0 now -> execute this plan -> bump pubspec to 0.3.1 -> publish again.

## RALPLAN-DR Summary

### Principles

1. **First-screenful hook.** A developer scanning pub.dev or GitHub must understand the value prop within the visible viewport (~20 lines of rendered markdown) without scrolling.
2. **Search-term alignment.** Every keyword slot (pubspec topics, GitHub topics, repo description, README H1 area) must target what developers actually type when looking for this type of widget -- not author vanity terms.
3. **Copy-paste onboarding.** Every code block in the README and example must compile, run, and demonstrate a working feature. No "exercise for the reader" gaps.
4. **Parallel-safe file ownership.** The three axes must be executable by independent agents with zero merge conflicts. Any shared file gets an explicit ordering constraint.
5. **Minimal churn.** Preserve the README's existing deep-dive content (behavior details, FAQ). Restructure the top half; leave the bottom half stable.

### Decision Drivers (top 3)

1. **pub.dev discovery.** The package has zero pub.dev points for "popularity" (not yet published). First impressions from topics, description, and README quality determine whether it gets bookmarked or skipped.
2. **v0.3.0 feature visibility.** `abortSnap()` and `pinnedDivider*` are the main reasons to upgrade from v0.2.0. If the README does not surface them, users won't know they exist.
3. **Zero-friction example.** The example app is what `flutter create --sample` users see first. It must demonstrate the headline feature (pinnedDivider) without complicating the simple path.

## ADR (Architectural Decision Record)

### Decision
Restructure README top-half for first-screenful hook, replace 3/5 pub.dev topics for search alignment, update example to showcase v0.3.0's pinnedDivider, and ship as v0.3.1 docs-only release after publishing v0.3.0.

### Drivers
1. pub.dev first-impression optimization (README hook + topics).
2. v0.3.0 feature visibility in docs and example.
3. Parallel-safe execution across 3 independent agents.

### Alternatives considered
- **Additive README patch (A-alpha):** Rejected -- pushes fold further down, fails hook principle.
- **README + separate DEEP_DIVE.md (A-gamma):** Rejected -- pub.dev cannot render secondary files.
- **Leave example untouched (C-alpha):** Rejected -- misses the chance to showcase pinnedDivider at zero cost.
- **Add second example file (C-gamma):** Rejected -- pub.dev only shows main.dart; duplicates README.
- **Add doc/WHY.md (C3-beta):** Rejected -- invisible on pub.dev, redundant with existing README sections.
- **Amend v0.3.0 instead of shipping v0.3.1:** Rejected -- blocks feature delivery, bad pub.dev practice.

### Why chosen
Beta variants for README structure and example update offer the best ratio of impact to diff size. The "publish v0.3.0 first, then v0.3.1" sequence keeps the API release clean and the docs release semantic. Topic changes target actual developer search patterns rather than author-descriptive terms.

### Consequences
- README top half will have a larger diff vs v0.3.0. Git blame for those lines resets.
- 3 pub.dev topics change, which may briefly affect search ranking until re-indexed (typically <24h).
- Example app visual output changes deliberately: the divider is now pinned (stays visible when bar compresses) vs v0.3.0's scroll-away `SliverToBoxAdapter` divider. This is intentional -- the example now demonstrates the `pinnedDividerHeight` feature.

### Follow-ups
- After v0.3.1 publish, monitor pub.dev search ranking for "flutter search bar sliver" and "telegram search bar flutter" queries.
- If users request a deeper "Why not SliverAppBar?" doc, add `doc/WHY.md` in a future release.
- Consider adding screenshots/gif showing the specific SliverAppBar vs this package difference for a future README enhancement.

## Risks + Mitigations

### Risk 1: `gh` CLI not authenticated on executor machine

Step 2B uses `gh api` for GitHub topics and About description updates. If `gh` is not authenticated, those calls silently fail or error.

**Mitigation:** Step 2B opens with a pre-check: `gh auth status 2>&1 | grep -q "Logged in"`. If the check fails, the task blocks with a message telling the operator to run `gh auth login` and rerun. No silent failure.

### Risk 2: pub.dev topic rejection

Some proposed topics (especially `telegram`) could be flagged as a brand term and rejected by pub.dev's topic validator.

**Mitigation:** Named fallback for each disputed topic:
- If `telegram` is rejected, fall back to `collapsing-header`.
- If `animation` is rejected (unlikely), fall back to `gesture`.

The executor runs `flutter pub publish --dry-run` after changing topics and checks stderr for topic-rejection warnings. If any, substitute the fallback and re-run.

### Risk 3: `flutter pub login` credentials not present
