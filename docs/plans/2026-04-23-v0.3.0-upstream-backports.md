## RALPLAN-DR Summary

### Principles

1. **Non-breaking over convenience.** All new parameters are optional with null/false defaults. Existing v0.2.0 callers compile and behave identically without changes.
2. **One canonical pattern per concern.** Each behavioral addition (abort snap, pinned divider) maps to exactly one public API surface, not two competing approaches.
3. **Upstream parity first, then generalize.** The implementation mirrors the proven upstream logic (same math, same branches), then wraps it in configurable parameters for reuse.
4. **TDD rigor.** Every task starts with a failing test, then the minimal implementation to pass it. No speculative code.
5. **Minimal diff surface.** Changes touch only the files that must change. No cascading refactors.

### Decision Drivers (top 3)

1. **UX parity with Telegram iOS.** The two backported behaviors (abort inflight snap on re-touch, pinned 1px divider) directly close visual/tactile gaps versus TG.
2. **Back-compat with v0.2.x callers.** Zero migration needed for anyone using the existing API. New params default to null/off.
3. **Test coverage.** Every new code path has at least one test with explicit `expect` assertions. Baseline 42 tests must remain green throughout.

### Viable Options

#### Item A — pointerDown abort

**Option A-alpha: Public `abortSnap()` on `SliverSnapController` + `onPointerDown` in `SliverSnapView` Listener.**
- Pro: Clean public API. Callers using primitives (not `SliverSnapView`) can also call `abortSnap()` from their own `Listener.onPointerDown`.
- Pro: Mirrors upstream `_abortInflightSnap()` 1:1.
- Con: None identified.

**Option A-beta: Internal-only abort, no public method. `SliverSnapView` calls `scrollController.jumpTo(offset)` directly in its own `onPointerDown`.**
- Pro: Slightly fewer lines (no new method on controller).
- Con: Primitive-level callers cannot use it. Breaks the controller-as-single-coordination-point pattern. The view would need to reach into controller internals (`_isSnapping`), violating encapsulation.
- **INVALIDATED.** Encapsulation violation makes this a non-starter; the controller owns snap state.

**Decision: A-alpha.** Single viable option after elimination.

#### Item B — pinned divider in delegate

**Option B-alpha: Two primitive params `pinnedDividerHeight: double?` + `pinnedDividerColor: Color?`.**
- Pro: Simple, zero allocation. Matches upstream's `Container(height: h, color: c)` exactly.
- Pro: No widget tree overhead — just a `Container`.
- Con: Cannot render gradients, shadows, or custom widgets as the divider.

**Option B-beta: One widget slot `pinnedDivider: Widget?`.**
- Pro: Maximum flexibility — any widget as divider.
- Con: Caller must manually size it. The delegate cannot compute `dividerHeight` from a widget's intrinsic dimensions at construction time (delegate is not a `RenderObject`). Would require a separate `pinnedDividerHeight` param anyway to compute `minExtent`/`maxExtent`, defeating the "one slot" simplicity.
- Con: `shouldRebuild` cannot compare arbitrary widgets by identity (same problem as `child`/`builder` — callers rebuild every frame). Would need either a nullness-only check (coarse) or force callers to provide a stable key.

**Option B-gamma: Both primitives + widget slot (let user pick).**
- Pro: Covers both simple and advanced cases.
- Con: Three new params for one concern. Interaction semantics are complex: "if widget is set, ignore height/color? or use height from param + widget for content?" Overdesigned for a 1px line.

**Decision: B-alpha (two primitives).** Rationale:
- The upstream pattern is literally `Container(height: 1, color: dividerColor)`. The 99% use case is a 1px colored line.
- `pinnedDividerHeight` is required for extent math regardless — no way around it.
- Callers needing a gradient divider can use the existing `builder` param on the delegate and render it themselves, or use the `SliverSnapView.divider` (free-floating `SliverToBoxAdapter`) with a custom widget.
- Adding a widget slot can be done in a future minor version if demand materializes; removing a widget slot is a breaking change. Starting conservative is safer.

### Deprecated typedef aliases

**Decision: Keep all three `@Deprecated` aliases (`SnapSearchBarController`, `SnapSearchBarView`, `DefaultSnapSearchBarRow`) through v0.3.x. Schedule removal for v0.4.0.** v0.2.0 shipped the same day; removing aliases in the next minor release gives callers zero grace period. Update the `@Deprecated` message strings from "removed in v0.3.0" to "removed in v0.4.0". Add a CHANGELOG note.

Two UX improvements from the upstream `customer-im-client` project need backporting to this generic package:

1. **Item A — pointerDown abort inflight snap.** When a snap animation is running (140ms), a new touch should immediately cancel it so the user's drag starts cleanly.

TG iOS keeps a 1px line always pinned under the navbar, achieved by rendering the divider inside the delegate and setting `minExtent = dividerHeight` when not searching.

```dart
void maybeSnapOnPointerUp() {
  _assertNotDisposed();
  if (_isSnapping) return;
  if (!scrollController.hasClients) return;

  final pixels = scrollController.position.pixels;
  if (pixels <= 0 || pixels >= totalHeight) return;

  final target = pixels < totalHeight / 2 ? 0.0 : totalHeight;
  _isSnapping = true;
  final gen = ++_snapGeneration;  // capture this snap's generation
  scrollController.jumpTo(pixels);
  scrollController
      .animateTo(target, duration: snapDuration, curve: snapCurve)
      .then(
        (_) {},
        onError: (Object err, StackTrace st) {
          if (err is FlutterError) return;
          throw err;
        },
      )
      .whenComplete(() {
        // Only clear if this is still the active snap. A subsequent
        // abortSnap() + new snap would bump _snapGeneration; stale
        // whenComplete from the aborted animation must NOT clear the
        // new snap's guard.
        if (!_disposed && _snapGeneration == gen) _isSnapping = false;
      });
}
```

Add `abortSnap()` (after `resetSnapFlag()`, before `dispose()`):

```dart
/// Immediately cancels any in-progress snap animation started by
/// [maybeSnapOnPointerUp]. Call from a `Listener.onPointerDown` to
/// ensure the user's new touch is not fighting the tail of a 140ms
/// snap.
///
/// Safe to call when not snapping, disposed-scroll, or no clients
/// — each case is a silent no-op (except post-dispose, which asserts
/// in debug mode).
void abortSnap() {
  _assertNotDisposed();
  if (!_isSnapping) return;
  if (scrollController.hasClients) {
    scrollController.jumpTo(scrollController.position.pixels);
  }
  _snapGeneration++;  // invalidate in-flight whenComplete
  _isSnapping = false;
}
```

## Risk Section

### Risk 1: Callers passing `divider` to `SliverSnapView` see changed behavior

**Assessment:** No change. The `divider` param still wraps in `SliverToBoxAdapter` at index 1 of the slivers list. `pinnedDividerHeight`/`pinnedDividerColor` are separate params that affect the delegate, not the slivers list. Both are null by default. Existing callers pass neither `pinnedDividerHeight` nor `pinnedDividerColor`, so their delegate extents and build output are identical.

**Mitigation:** Explicit test (Task B2) verifying both mechanisms coexist.

### Risk 2: `pinnedDividerHeight != null` changes `minExtent` for direct delegate users

**Assessment:** Callers using `SliverSnapSearchBarDelegate` directly (not through `SliverSnapView`) who add `pinnedDividerHeight` will see `minExtent` change from 0 to `pinnedDividerHeight`. This is the intended behavior. Callers who do NOT pass the new param see null default, and `minExtent` stays 0. No migration needed.

**Mitigation:** Null default + CHANGELOG migration note saying "no breaking changes."

### Risk 3: Abort-then-resnap race — stale `whenComplete` corrupts new snap's `_isSnapping`

**Assessment:** Without the snap-generation counter, the following abort-then-resnap-within-same-event sequence is a real bug:

1. `maybeSnapOnPointerUp()` -- `_isSnapping = true`, starts `animateTo` (snap 1).
2. `abortSnap()` -- `jumpTo(pixels)`, `_isSnapping = false`.
3. New `maybeSnapOnPointerUp()` -- `_isSnapping = true`, starts new `animateTo` (snap 2).
4. Old `animateTo` (snap 1) `.whenComplete` fires on the microtask queue -- sets `_isSnapping = false`, **corrupting** snap 2's guard.

Without the counter, a new snap's `_isSnapping = true` guard can be cleared by the aborted animation's stale `whenComplete`, enabling a 2nd concurrent `animateTo` on the next pointer-up.

`jumpTo` itself is safe (synchronously cancels running animations, same pattern as inside `maybeSnapOnPointerUp`). The `.then(onError:)` handler swallows `FlutterError` lifecycle noise. The real hazard is the `whenComplete` timing.

**Mitigation:** A monotonic `_snapGeneration` counter is incremented by both `maybeSnapOnPointerUp` (pre-animate) and `abortSnap`. The `whenComplete` callback captures the generation at call-time and only writes `_isSnapping = false` when `_snapGeneration == gen`. A stale callback from an aborted animation sees a mismatched generation and is a no-op. Explicit test (`abort then restart snap: old whenComplete does not clear new _isSnapping`) verifies this.

## ADR

**Decision:** Backport two upstream improvements as additive, non-breaking changes in v0.3.0.

**Drivers:** UX parity with TG iOS, zero-migration for v0.2.x callers, upstream-proven implementation.

**Alternatives considered:**
- Item A beta (internal-only abort): invalidated due to encapsulation violation.
- Item B beta (widget slot): invalidated due to needing height param anyway + `shouldRebuild` complexity.
- Item B gamma (both primitives + widget): invalidated as overdesigned for a 1px line.

**Why chosen:** Minimal API surface (one public method for A, two optional params for B) with maximum upstream fidelity and zero breaking changes.

**Consequences:** Callers who want a custom widget as the pinned divider must use the `builder` param or compose primitives directly. If demand materializes, a `pinnedDivider: Widget` slot can be added in a future minor.

**Follow-ups:**
- v0.4.0: Remove `@Deprecated` typedef aliases.
- Consider exposing `abortSnap()` in the `SliverSnapView` API for callers who need programmatic abort (currently only wired to `onPointerDown`).
