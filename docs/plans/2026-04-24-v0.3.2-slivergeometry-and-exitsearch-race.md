# v0.3.2 Patch Release Plan — SliverGeometry layoutExtent assertion + exit-search snap race

**Date:** 2026-04-24
**Baseline:** v0.3.1 (tag `d584d39`), 62 tests pass, analyze clean, pub dry-run 0 warnings
**Upstream ref:** `~/Documents/customer-im-client/` branch `origin/dev`

## RALPLAN-DR Summary

### Principles

1. **Framework contract fidelity** — delegate output height must always satisfy `paintExtent >= layoutExtent` as computed by `SliverPersistentHeader`.
2. **Single formula source of truth** — use the framework's own `(maxExtent - shrinkOffset).clamp(minExtent, maxExtent)` formula for height, never a mathematically-equivalent but floating-point-divergent path.
3. **Symmetry between branches** — early-return and normal branches must have structurally parallel wrapping (SizedBox + ClipRect) so maintenance of one informs the other.
4. **No API surface changes** — patch release. Zero new public methods, zero signature changes.
5. **TDD commit discipline** — each fix ships with a test that discriminates pre-fix (fails) from post-fix (passes).

### Decision Drivers (top 3)

| # | Driver | Weight |
|---|--------|--------|
| 1 | Eliminate real-device SliverGeometry assertion (user-visible log spam, 142 entries / 3K lines) | Critical |
| 2 | Close the exit-search `_isSnapping` race window that swallows the first post-exit snap | High |
| 3 | Minimize diff surface for a patch release (no regressions, no refactors) | High |

## Decision 1 — Ship both bundles in v0.3.2, or split?

**Pick: (alpha) Both in v0.3.2.**

Rationale:
- Both are small correctness fixes (zero API changes, 2 files touched).
- Bundle A is 3 upstream commits but they all target the same function (`build()` in `sliver_snap_search_delegate.dart`), total ~15 lines of production code.
- Bundle B is a single line (`_isSnapping = false;`) plus one optional line (`_snapGeneration++;`).
- Shipping separately doubles release overhead for no isolation benefit — both are independent and cannot conflict.

Invalidated alternatives:
- **(beta) Only A:** Would leave the exit-search race open when both are trivial. No justification for delay.
- **(gamma) Only B:** Would ship a one-line race fix while leaving a user-visible log-spam bug open. Bundle A is the higher-severity item.

## Decision 2 — Bundle A: normal branch wrap — where exactly?

**Pick: (gamma) SizedBox outside ClipRect, matching upstream structure.**

Current our normal branch (line 230-237):
```dart
return ClipRect(
  child: _SnapSearchBarScope(
    ...,
    child: content,   // content = Column or body
  ),
);
```

After fix:
```dart
final expectedH = (maxExtent - shrinkOffset).clamp(minExtent, maxExtent);
return SizedBox(
  height: expectedH,
  child: ClipRect(
    child: _SnapSearchBarScope(
      ...,
      child: content,
    ),
  ),
);
```

Rationale:
- Upstream uses `SizedBox -> ClipRect -> Column`. Our `_SnapSearchBarScope` is an InheritedWidget wrapper that doesn't affect layout, so `SizedBox -> ClipRect -> _SnapSearchBarScope -> content` is structurally equivalent.
- SizedBox OUTSIDE ClipRect ensures the sliver's child reports `expectedH` to the framework before ClipRect can interfere with intrinsic sizing.
- (alpha) wrapping outside ClipRect is identical to (gamma) in our tree since `_SnapSearchBarScope` is purely inherited data — so gamma = alpha in practice.

Invalidated:
- **(beta) Inside ClipRect:** Functionally equivalent to option gamma, but gamma matches upstream's `SizedBox → ClipRect → Column` structure exactly, so reviewers comparing against upstream can verify parity by inspection. No technical blocker; purely parity preference.

## Decision 3 — Bundle A: pinnedDividerHeight == null path — wrap or leave?

**Pick: (alpha) Wrap every path (null and non-null) with framework-formula SizedBox.**

Rationale:
- Wrapping every path eliminates floating-point divergence in ALL branches. The null-divider early-return's `SizedBox(height: totalHeight * ratio)` computes `totalHeight * (1 - shrinkOffset/totalHeight)` — 3 FP ops (divide, subtract, multiply), each with rounding — whereas the framework formula `(maxExtent - shrinkOffset).clamp(minExtent, maxExtent)` = `totalHeight - shrinkOffset` is a single subtraction. The two paths differ by 1-2 ULPs in practice, producing the same 5e-15 dp-class assertions as the normal branch. D3 (alpha) closes all three sites, not just the obvious ones.
- Cost is one extra `SizedBox` wrapper. Consistency eliminates an entire class of future bug reports.
- Every return path from `build()` gets a `SizedBox(height: expectedH)` — single audit rule.

Invalidated:
- **(beta) Only non-null:** Saves one wrap site but leaves the null path vulnerable to the same floating-point assertion. Not worth the asymmetry.

## Decision 4 — Bundle B: where to clear `_isSnapping`?

**Pick: (alpha) + bump `_snapGeneration`.**

Add at top of `restorePreSearchOffset()`:
```dart
void restorePreSearchOffset() {
  _assertNotDisposed();
  _isSnapping = false;      // <-- NEW: pre-clear (symmetric with save)
  _snapGeneration++;         // <-- NEW: invalidate stale whenComplete
  _version++;
  _restoreInternal(_version);
}
```

Rationale for `_isSnapping = false`:
- `savePreSearchOffset()` already clears `_isSnapping` (line 150). `restorePreSearchOffset()` does not. This is the exact asymmetry upstream fix closes.
- Without pre-clear: `restorePreSearchOffset` calls `jumpTo` (via `_restoreInternal`) which interrupts an in-flight `animateTo`. The `whenComplete` runs on a later microtask. In the window between `jumpTo` and `whenComplete`, a fresh `pointerUp` hitting `maybeSnapOnPointerUp` sees `_isSnapping == true` and returns early — snap is swallowed.

Rationale for bumping `_snapGeneration`:
- **Yes, bump.** Matches `abortSnap()` convention (line 262-263): any code path that sets `_isSnapping = false` and may interrupt an in-flight animation should bump `_snapGeneration` to invalidate the stale `whenComplete`. This prevents the stale callback from setting `_isSnapping = false` at an arbitrary future time (harmless now but fragile under future changes).
- `savePreSearchOffset` does NOT currently bump `_snapGeneration`, but it should be considered a pre-existing gap. For this patch we only fix the reported bug (`restorePreSearchOffset`). A follow-up can symmetrize `save` too if desired.

  'restorePreSearchOffset clears isSnapping so next pointerUp is not swallowed',

    final ctrl = SliverSnapController(scrollController: scroll);

## Open Questions for Architect / Critic

1. **Should `savePreSearchOffset` also bump `_snapGeneration`?** Currently it clears `_isSnapping` but does NOT bump `_snapGeneration`. This means a stale `whenComplete` from a pre-save animation could fire and harmlessly set `_isSnapping = false` (already false). It is safe but asymmetric with `abortSnap()` and the proposed `restorePreSearchOffset` fix. Is this worth addressing in v0.3.2 or defer to a future release?

2. **The existing `early-return continuity` test (line 161-178) uses `find.byType(SizedBox).first` to locate the height.** After wrapping the null-divider early-return in `SizedBox(height: expectedH)`, the outer SizedBox is now the "first" one found. Since `expectedH` at shrinkOffset=52 with null pinnedDivider equals `totalHeight * (1 - 52/56)` = `totalHeight * ratio` (maxExtent = totalHeight, minExtent = 0), the numeric assertion still holds. But should we adjust the finder to be more specific (e.g. find a SizedBox with a specific height) for robustness against future widget tree changes?

3. **Test harness approach: `delegate.build()` direct call vs full `CustomScrollView` pump.** The existing tests call `delegate.build(context, shrinkOffset, false)` directly, which does NOT exercise the framework's `layoutExtent` computation. The SliverGeometry assertion only fires when the framework runs `performLayout` on the `RenderSliverPersistentHeader`. The new T1/T2 tests MUST use a full `CustomScrollView` + `SliverPersistentHeader` pump with `jumpTo(offset)` to actually trigger the framework's geometry validation. Confirm this is the correct approach and that the existing `_pumpDelegate` helper should NOT be reused for these tests.
